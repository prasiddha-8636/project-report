\chapter{FEASIBILITY ANALYSIS}

\section{Overview}
This chapter examines whether the proposed system is practical to build and run. We work through the data requirements, storage costs, and computational loads for each stage of the pipeline, then describe the validation strategy.

\section{Elevation Data and Database Generation}
\label{sec:elevation}

\subsection{Choice of DEM}
A Digital Elevation Model stores terrain height as a grid of cells. We need this grid to reconstruct 3D mountain horizons for matching.

We selected Copernicus GLO-30, which covers the globe at $30\,\text{m}$ resolution and was released by the European Space Agency in 2020 \cite{Franks2023, Trevisani2023}. In rugged alpine terrain, GLO-30 has a measured root-mean-square vertical error of about $2.34\,\text{m}$ \cite{Franks2023}. Comparative studies in high-relief mountain ranges confirm that it provides a more consistent surface than older alternatives like AW3D30 \cite{Trevisani2023}.

We considered two other models but found them unsuitable:
\begin{itemize}
    \item \textbf{SRTM and NASADEM:} Both have data gaps in steep areas caused by radar shadow and layover \cite{Franks2023}.
    \item \textbf{FABDEM:} Designed to remove forest canopy and buildings from the GLO-30 base. Since the high-altitude Khumbu terrain consists almost entirely of rock and ice with negligible vegetation or human structures, the canopy-removal features of FABDEM offer no functional advantage over the base model for this application.
\end{itemize}

\subsection{Data Volume and Computational Requirements}
The study area covers the main trekking corridors of Khumbu ($40\,\text{km} \times 30\,\text{km}$). To prevent horizons from being clipped at the edges, we add a $50\,\text{km}$ buffer around the target area—matching the maximum ray-casting distance to ensure all visible peaks are included—giving a total bounding box of roughly $140\,\text{km} \times 130\,\text{km}$. At $30\,\text{m}$ spacing, this contains about 20 million elevation cells and requires $50$--$80\,\text{MB}$ of storage.

The reference database is built by placing viewpoints on an $80 \times 60$ grid spaced $500\,\text{m}$ apart (chosen as a balance between localization resolution and database size, sufficient for identifying major trail sections), giving $4{,}800$ viewpoints. At each one, we cast 1,440 rays at $0.25^{\circ}$ intervals:
\[
360^{\circ} / 0.25^{\circ} = 1440 \text{ rays.}
\]

Each ray steps through the DEM out to a maximum distance of $50\,\text{km}$, requiring roughly $1{,}667$ elevation lookups per ray. So one profile costs:
\[
1440 \times 1667 \approx 2.4 \times 10^6 \text{ elevation checks.}
\]

Across all 4,800 viewpoints:
\[
4800 \times 2.4 \times 10^6 \approx 11.5 \times 10^9 \text{ total checks.}
\]

This is large but straightforward. Array-based processing on a modern computer finishes the entire generation in minutes \cite{Steger2022}.

\subsection{Database Storage}
\label{sec:dem-storage}
Each profile is stored as 1,440 single-precision floats (4 bytes each):
\[
1440 \times 4 = 5760 \text{ bytes} \approx 5.6\,\text{KB per profile.}
\]

We store three versions of each profile, computed at different maximum search radii (global, local, and restricted) to handle varying visibility. All three tiers are stored per viewpoint and available for matching:
\[
5.6\,\text{KB} \times 3 \approx 16.9\,\text{KB per viewpoint.}
\]

For all 4,800 viewpoints:
\[
4800 \times 16.9\,\text{KB} \approx 79\,\text{MB.}
\]

This fits comfortably in system memory. A direct in-memory search is sufficient---no external database or indexing library is needed.

\section{Two-Stage Matching Pipeline}
The matching pipeline has two stages: a fast global search followed by fine alignment.

\subsection{Step 1: FFT-based Sliding-Window Search}
A query photograph produces a partial skyline of length $m$ (for example, $m = 240$ points for a $60^{\circ}$ field of view at $0.25^{\circ}$ resolution). Since the camera heading is unknown, this partial profile must be compared against every angular offset of each $4{,}800$ reference profiles ($N = 1440$ points each).

A naive sliding-window comparison costs $O(N \times m)$ per profile. We speed this up with FFT-based cross-correlation, which computes the full Euclidean distance profile in $O(N \log N)$ time \cite{mueen2024mass}. We deliberately skip z-normalization to preserve the absolute elevation values in the signal.

The per-profile operation counts illustrate the speedup:
\begin{itemize}
    \item \textbf{Brute force:}
    \[
    N \times m = 1440 \times 240 = 345{,}600 \text{ operations.}
    \]
    \item \textbf{FFT-based:}
    \[
    N \log_2 N \approx 1440 \times 10.49 \approx 15{,}106 \text{ operations.}
    \]
\end{itemize}

Across all 4,800 profiles, brute force requires about 1.66 billion operations; the FFT approach needs roughly 73 million---a 23-fold reduction. Based on the documented throughput of the HORAYZON engine \cite{Steger2022} and the complexity of the MASS algorithm \cite{mueen2024mass}, this first stage is projected to complete in under one second on standard consumer hardware.

\subsection{Step 2: DTW Refinement}
The first stage produces 50--100 candidate viewpoints. This shortlist represents the top 1--2\% of the 4,800 global viewpoints by L2 distance, providing a sufficient subset for elastic alignment while maintaining low computational overhead. Camera focal length and tilt can stretch or compress the captured skyline in ways that Euclidean distance cannot handle. We therefore re-rank the candidates using Dynamic Time Warping (DTW) \cite{Rakthanmanon2013}, which is consistent with established terrain-matching pipelines \cite{pan2020camera}.

Standard DTW has $O(m^2)$ complexity \cite{Rakthanmanon2013}. We apply two optimizations:
\begin{enumerate}
    \item \textbf{Sakoe-Chiba band:} The warping path is confined to a diagonal window of width $w$. Setting $w = 24$ (10\% of the query length, a standard heuristic for time-series matching \cite{Rakthanmanon2013}) reduces the cost per candidate to $O(m \times w)$.
    \item \textbf{Early abandoning and pruning:} Paths exceeding the current best distance are terminated immediately \cite{Rakthanmanon2013}.
\end{enumerate}

Per candidate:
\[
m \times w = 240 \times 24 = 5760 \text{ operations.}
\]

Evaluating 100 candidates costs $576{,}000$ operations---negligible on any modern processor.

\section{Extraction and Validation}

\subsection{Sky Segmentation Model}
We use a U-Net with a MobileNet-V3 backbone for sky segmentation \cite{long2015fully, howard2019mobilenetv3}. A Canny edge detector \cite{Canny1986} serves as a fallback if the neural network fails. Other recent models such as YUNet \cite{Yang2025} provide useful benchmarks for comparison.

The model is pre-trained on GeoPose3K \cite{Brejcha2017}, which contains over 3,000 alpine mountain images with known camera poses. Since those images come from the European Alps, we fine-tune on synthetic Khumbu terrain generated from the DEM and augmented with fog textures \cite{tomesek2022cross}.

\subsection{Ground-Truth Verification}
For real-world testing, we use panoramas from the 2014 Google Street View Trekker campaign along the Lukla--Everest Base Camp trail \cite{google2015streetview}. This collection has over 45,000 panoramas with verified GPS coordinates.

We do not match against full panoramas directly. Instead, we cut each panorama into perspective photographs (e.g., 12 crops at $30^{\circ}$ intervals, each covering a $60^{\circ}$ horizontal field of view) and run the pipeline on these individual frames. The validation strategy includes:
\begin{itemize}
    \item \textbf{Preventing data leakage:} Train/test splits are made at the panorama level so that overlapping crops from the same physical location do not appear in both sets.
    \item \textbf{Correcting distortion:} Perspective crops are re-projected from equirectangular format into planar views.
    \item \textbf{Weather categories:} Test images are grouped by visibility to evaluate each database weather tier.
\end{itemize}

\section{Implementation and Resources}

\subsection{System Specifications}
The full reference database is about $79\,\text{MB}$. We store horizon profiles in a structured in-memory index and search them directly via FFT, without relying on external vector-indexing libraries.

With a $500\,\text{m}$ grid spacing, localization accuracy is measured by correct grid-cell assignment. The heading estimation is constrained by the $\pm 0.25^{\circ}$ angular resolution of the database profiles, which represents the theoretical resolution limit of the sampling interval.

\subsection{Project Timeline}
DEM processing, database generation, and matching are all computationally light and fit within a 60-day schedule. Most of the effort goes into downloading and preparing the Street View validation set, fine-tuning the segmentation model, and running the evaluation. To avoid API rate limits, we process a representative subset of several hundred panoramas, yielding several thousand test frames.

\section{Conclusion}
The computational, storage, and data requirements of the proposed system are manageable within the available resources and timeline.
